# Sorting Algorithms: Experimental Results and Analysis

## Overview

This document analyzes the experiments implemented in [algorithms.c](algorithms.c) and summarizes observed timings recorded in [results.csv](results.csv). The goal is to compare algorithmic behavior across input sizes and input structures, and to check whether measured behavior matches theoretical expectations.

**Algorithms tested:** bubble sort, quick sort, insertion sort, selection sort, merge sort, radix sort (see [algorithms.c](algorithms.c) for implementations).

## Experiment Structure (from algorithms.c)

- **Input arrays**: integer arrays generated by helper functions in `algorithms.c`:
  - `generateIntArray(size)`: uniform random integers in range [0, size]
  - `generateReverseSortedIntArray(size)`: strictly descending sequence
  - `generatePartiallySortedIntArray(size, factor)`: initially sorted then `size/factor` random swaps (factor values tested: 10, 30, 50, 70, 90)

- **Sizes and runs** (as exercised in the code):
  - size = 10, runs = 1,000,000
  - size = 50, runs = 1,000,000
  - size = 100, runs = 1,000,000
  - size = 1000, runs = 100
  - size = 10000, runs = 10

- **Variants**: `reversed`, `random`, `partially sorted X%` (where X is 10, 30, 50, 70, 90)

- ** timing**: each run measures wall-clock CPU time using `clock()` in C; times are averaged over the specified number of runs and written to `results.csv`.

## Key Observations from results.csv

Representative datapoints (extracted from [results.csv](results.csv)): 

- Small arrays (10–100) with very large run counts: reported times are in the sub-millisecond range and often identical across variants (e.g., many entries show ~0.001–0.05 ms). This indicates timer resolution and per-call overhead dominate for tiny arrays.

- For size = 1000 (100 runs):
  - Quick sort: `random` 0.160 ms, `reversed` 3.215 ms
  - Merge sort: `random` 0.214 ms, `reversed` 0.124 ms
  - Radix sort: `random` 0.103 ms, `reversed` 0.066 ms
  - Bubble/insertion/selection: significantly slower (bubble ~3.1 ms reversed, insertion ~1.2 ms random, selection ~2.3 ms random)

- For size = 10000 (10 runs):
  - Quick sort: `random` 2.140 ms, `reversed` 312.116 ms
  - Merge sort: `random` 2.778 ms, `reversed` 1.481 ms
  - Radix sort: `random` 1.010 ms, `reversed` 0.856 ms
  - Bubble/insertion/selection: enormous times (bubble ~310–361 ms, insertion ~118–235 ms, selection ~206–239 ms)

These show a strong divergence between O(n^2) algorithms and O(n log n)/linear-ish algorithms as size grows.

## Analysis — Do the observations match expectations?

- Overall: Yes — the measured patterns align with theoretical complexity and standard practical behavior:
  - O(n^2) algorithms (bubble, selection, insertion) scale poorly: nearly negligible for tiny arrays but rapidly become impractical at n=1000 and catastrophic at n=10000. The CSV shows bubble/insertion/selection times growing by orders of magnitude between 1000 and 10000, as expected.
  - O(n log n) algorithms (merge sort, quick sort average-case) and radix sort (linear in digit count) remain fast at larger sizes.

- Important specific matches:
  - Insertion sort is very fast on nearly sorted arrays (e.g., for size=10000 partially sorted 90% it's much faster than on random), matching its O(n + k) practical behavior.
  - Merge sort timings are steady across input variants (random, reversed, partially sorted) — consistent with guaranteed O(n log n) independent of input order.
  - Radix sort performs exceptionally well on these integer datasets (limited digit range tied to `size`) — consistent with linear-ish digit-by-digit counting sort.

- Notable deviation / implementation-driven behavior:
  - Quick sort shows very large times on `reversed` and some `partially sorted` inputs (e.g., 312 ms vs 2.14 ms for size=10000). This is an implementation artifact: the pivot is chosen as the last element (see `partitionQuickSort` in [algorithms.c](algorithms.c)). Choosing the last element as pivot leads to worst-case O(n^2) behavior on already sorted or reverse-sorted data. The observed huge times for reversed inputs are consistent with that flaw.
  - Selection sort times are essentially invariant to input ordering (selection always scans for min), which matches theoretical expectations and the measured data.

- Measurement caveats that explain some small-scale artifacts:
  - Timer resolution and averaging: for sizes 10–100 with 1,000,000 runs, many lines show identical tiny values (0.001 ms). The `clock()` resolution and per-run overhead (malloc/free, function call, RNG) dominate, so these microbenchmarks are noisy and not informative. Use fewer runs or higher-resolution timers for small arrays.
  - Memory allocation overhead: each run generates a fresh array with `malloc` and `free`. For small arrays and massive repetition, allocation overhead can dominate. For larger arrays, the sort cost dominates and allocation overhead is proportionally smaller.
  - Single-threaded timing: all tests are single-threaded; system noise, OS scheduling, and caching effects can add variance — especially where runs are few (n=100 or n=10).

## Recommendations & Possible Improvements

- Fix quick sort pivot selection to avoid worst-case on sorted inputs:
  - Use median-of-three, random pivot, or introsort (fallback to heap sort) to guarantee robust performance.
- Reduce measurement noise for small arrays:
  - Reuse buffers instead of malloc/free inside each run.
  - Use fewer runs for small sizes or increase array size for timing stability.
  - Use a higher-resolution timer (e.g., `clock_gettime(CLOCK_MONOTONIC, ...)`) and record standard deviation in addition to mean.
- Report more statistics: record median, standard deviation, and min/max per configuration.
- For radix sort: test different value ranges (many duplicates, wide numeric ranges) to show where radix's digit/byte cost pays off or becomes expensive.
- Extend experiments: strings, floats, linked-lists, parallel implementations, and larger sizes (with careful sampling) to expand the study.

## Conclusion

The experimental data in [results.csv](results.csv) largely confirms theoretical expectations: O(n^2) sorts are fine for very small arrays but fail on larger inputs, while merge/radix/average quick sort are fast. A prominent implementation issue (pivot choice) causes quick sort to degrade to quadratic time on sorted/reversed inputs — this is the main actionable finding.
